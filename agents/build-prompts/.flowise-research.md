# Flowise Codebase Analysis for Archon

> Reference: `~/Scripts/flowise-reference` (FlowiseAI/Flowise, cloned for study)
> Purpose: Extract UX and architecture patterns to inform Archon's visual builder

---

## 1. Node Palette UX

### What It Does
A floating sidebar panel (triggered by a "+" button) that presents all available node types organized by category and framework tag. Users search with fuzzy matching, browse categorized accordions, and drag nodes onto the canvas.

### Key File Paths
| File | Role |
|------|------|
| `packages/ui/src/views/canvas/AddNodes.jsx` | Main palette: search, tabs, categorized accordion, drag-start |
| `packages/ui/src/api/nodes.js` | Frontend API client for fetching node definitions |
| `packages/ui/src/store/reducers/canvasReducer.js` | Redux store holding `componentNodes` array |
| `packages/server/src/NodesPool.ts` | Server-side registry that scans and loads all node classes |

### Architecture Pattern
- **Server-driven node registry**: The server scans a `nodes/` directory at startup, instantiates each node class, and exposes them via `GET /api/v1/nodes`. The UI fetches the full list once on canvas mount and stores it in Redux (`SET_COMPONENT_NODES`).
- **Category + tag two-level grouping**: Each node definition carries a `category` string (e.g., "LLMs", "Tools", "Memory") and a `tags` array (e.g., `["LangChain"]`, `["LlamaIndex"]`). The palette uses MUI Tabs for tag filtering and MUI Accordion for category grouping.
- **Blacklist filtering per canvas type**: Agent canvas and chatflow canvas each define a blacklist of categories and per-category exceptions. This keeps the palette contextually relevant.
- **Fuzzy search with scoring**: A custom scoring algorithm (not a library) awards points for substring matches, word-boundary alignment, consecutive character streaks, and penalizes gaps. Results are sorted by score and re-grouped by category. 500ms debounce.
- **Drag-and-drop initiation**: `onDragStart` serializes the full node definition JSON into `dataTransfer`, which the canvas drop handler reads.

### How to Adapt for Archon
- Define an `IAgentNode` interface with `category`, `tags`, `icon`, `description`, `inputs`, `outputs` fields. Populate from a registry (could be a static manifest or API endpoint).
- Build a palette component with search + category accordion. Archon categories might be: "Agents", "Tools", "Models", "Memory", "Orchestration", "I/O".
- Implement contextual filtering: hide node types that don't apply to the current flow type (e.g., hide multi-agent orchestration nodes in a single-agent canvas).
- Use a standard fuzzy search library (like `fuse.js`) rather than rolling a custom scorer.

## 2. Drag-Drop Canvas

### What It Does
A React Flow–based infinite canvas where users visually wire nodes together. Supports zoom, pan, grid snapping, custom node rendering, typed connection validation, and a custom edge with a delete button.

### Key File Paths
| File | Role |
|------|------|
| `packages/ui/src/views/canvas/index.jsx` | Main `<ReactFlow>` setup, drop handler, connection handler, dirty tracking |
| `packages/ui/src/views/canvas/CanvasNode.jsx` | Custom node renderer (icon, label, inputs, outputs, context menu) |
| `packages/ui/src/views/canvas/ButtonEdge.jsx` | Custom edge with centered delete button |
| `packages/ui/src/views/canvas/StickyNote.jsx` | Annotation node type |
| `packages/ui/src/views/canvas/NodeInputHandler.jsx` | Renders input handles + inline parameter widgets on nodes |
| `packages/ui/src/views/canvas/NodeOutputHandler.jsx` | Renders output handles with type-based positioning |
| `packages/ui/src/store/context/ReactFlowContext.jsx` | Context wrapper exposing `deleteNode`, `duplicateNode`, `onNodeDataChange` |

### Architecture Pattern
- **Two custom node types**: `customNode` (CanvasNode) and `stickyNote`. One custom edge type: `buttonedge`.
- **Three-layer state management**:
  1. **React Flow internal state** via `useNodesState` / `useEdgesState` hooks (local to canvas).
  2. **Redux** for cross-component state: `isDirty`, `chatflow` metadata, `componentNodes`, dialog visibility.
  3. **React Context** (`ReactFlowContext`) for imperative operations: `deleteNode`, `duplicateNode`, `deleteEdge`, `onNodeDataChange`.
- **Drop handler flow**: On drop, reads `dataTransfer` JSON → projects screen coords to flow coords via `reactFlowInstance.project()` → generates unique node ID → calls `initNode()` to set defaults → appends to nodes array.
- **Connection validation**: Handles encode type info as `nodeId-type-paramName`. On connect, the handler checks `baseClasses` compatibility between source output and target input before creating the edge.
- **Dirty tracking**: Any node drag, connection change, or data edit dispatches `SET_DIRTY` to Redux. The canvas header shows an unsaved indicator and prompts on navigation.
- **Canvas config**: `minZoom: 0.1`, `snapGrid: [25, 25]` (toggleable), background dots (toggleable), `fitView` on load, delete key bound to `Delete` (disabled when dialogs open).

### How to Adapt for Archon
- Use `@xyflow/react` (React Flow v12+) with two custom node types: `agentNode` and `annotationNode`. One custom edge type with a delete button.
- Store canvas state in Zustand (simpler than Redux for flow state). Use React Flow's built-in `useReactFlow` hook instead of a custom context for imperative actions.
- Implement typed handles: define `baseClasses` on outputs (e.g., `["AgentOutput", "TextStream"]`) and `acceptClasses` on inputs. Validate on connect.
- Keep the drop handler pattern: serialize node template to `dataTransfer`, project coords, generate ID, initialize defaults.
- Add minimap (Flowise doesn't use one, but React Flow provides `<MiniMap />` out of the box).

## 3. Node Configuration

### What It Does
Each node type declares a typed parameter schema. When a user selects a node on the canvas, its parameters render inline on the node card and/or in an expanded dialog. The UI dynamically generates the correct input widget per parameter type.

### Key File Paths
| File | Role |
|------|------|
| `packages/components/src/Interface.ts` | Core interfaces: `INode`, `INodeParams`, `INodeOutputsValue`, `NodeParamsType` |
| `packages/components/nodes/[category]/[Name]/[Name].ts` | Individual node class with `inputs[]`, `outputs[]`, `credential`, `loadMethods` |
| `packages/server/src/NodesPool.ts` | Loads all node classes, resolves icons, builds registry |
| `packages/ui/src/views/canvas/NodeInputHandler.jsx` | 1300+ line component: maps `type` → widget, handles variable injection |
| `packages/ui/src/views/canvas/CanvasNode.jsx` | Renders node card, separates main vs `additionalParams` |
| `packages/ui/src/views/canvas/CredentialInputHandler.jsx` | Credential-specific input handling |

### Architecture Pattern
- **Declarative parameter schema**: Each node class defines an `inputs: INodeParams[]` array. Each param has: `name`, `label`, `type` (enum), `default`, `optional`, `description`, `options`, `loadMethod`, `acceptVariable`, `additionalParams`, `rows`, `step`, `placeholder`, etc.
- **Type-driven widget dispatch**: `NodeInputHandler` uses a large conditional block to map `INodeParams.type` to UI components:
  - `string/password/number` → `<Input>` or `<RichInput>`
  - `options` → `<Dropdown>`
  - `asyncOptions` → `<AsyncDropdown>` (calls `loadMethods` on server)
  - `code` → `<CodeEditor>` (syntax-highlighted)
  - `json` → `<JsonEditorInput>`
  - `file` → `<File>` upload
  - `boolean` → `<SwitchInput>`
  - `datagrid` → `<DataGrid>` table
  - `tabs` → `<Tabs>` with nested inputs
  - `array` → `<ArrayRenderer>`
  - `credential` → `<CredentialInputHandler>`
- **Two-zone rendering**: Parameters with `additionalParams: false` render inline on the node card. Those with `additionalParams: true` go into an expandable dialog (keeps the card compact).
- **Dynamic option loading**: Parameters with `type: 'asyncOptions'` and a `loadMethod` string trigger a server call to fetch options at runtime (e.g., listing available models from an API key).
- **Variable injection**: Many inputs support `{{nodeId.data.instance}}` template syntax, allowing one node's output to feed into another's config.

### How to Adapt for Archon
- Define an `IAgentNodeParam` interface: `{ name, label, type, default, required, options, description, group }`. Keep the type enum small initially: `text`, `number`, `select`, `code`, `json`, `boolean`, `credential`, `file`.
- Build a `<ParamRenderer>` component that dispatches on `type` → widget. Use a map/registry pattern instead of a long if-else chain.
- Split params into "primary" (shown on card) and "advanced" (shown in side panel or dialog). Use a `group` or `advanced: boolean` field.
- For async options (e.g., listing MCP tools from a connected server), define a `loadOptions` async function on the param that the renderer calls on mount.

## 4. Chatflow Testing

### What It Does
A floating chat panel within the canvas builder that lets users test their flow in real-time. Supports streaming (SSE) and non-streaming responses, displays source documents, tool usage, agent reasoning steps, and collects user feedback.

### Key File Paths
| File | Role |
|------|------|
| `packages/ui/src/views/chatmessage/ChatPopUp.jsx` | Floating panel wrapper, opened from canvas FAB button |
| `packages/ui/src/views/chatmessage/ChatMessage.jsx` | 1500+ line core: message state, send logic, streaming, display |
| `packages/ui/src/api/prediction.js` | Frontend API: `sendMessageAndGetPrediction`, streaming variant |
| `packages/server/src/routes/internal-predictions/index.ts` | Internal prediction routes (builder-only) |
| `packages/server/src/controllers/internal-predictions/index.ts` | Controller: builds chatflow, runs prediction, manages SSE |
| `packages/server/src/utils/SSEStreamer.ts` | Server-Sent Events connection manager |
| `packages/server/src/routes/predictions/index.ts` | Public prediction routes (deployed chatflows) |

### Architecture Pattern
- **Dual prediction endpoints**: Internal (`/api/v1/internal-prediction/{id}`) for builder testing and public (`/api/v1/prediction/{id}`) for deployed chatflows. Internal endpoints skip domain validation.
- **Streaming decision**: UI calls `GET /chatflows-streaming/{id}` to check if the flow supports streaming. If yes, uses `fetchEventSource` (SSE); if no, uses a standard POST with JSON response.
- **SSE event types**: `start`, `token` (incremental text), `sourceDocuments`, `usedTools`, `calledTools`, `agentReasoning`, `artifacts`, `fileAnnotations`, `metadata`, `error`. Each event type has a dedicated handler that updates message state.
- **Message state model**: Each message object holds: `id`, `message`, `type` (user/api), `sourceDocuments`, `usedTools`, `agentReasoning`, `artifacts`, `action` (for user confirmation buttons), `feedback`.
- **SSE connection management**: `SSEStreamer` on the server maintains a map of `chatId → response` objects. When the chatflow execution emits events, SSEStreamer writes them to the correct client connection.
- **Chat history**: Stored in `localStorage` keyed by chatflow ID. Allows continuity across page reloads during development.

### How to Adapt for Archon
- Build a `<TestPanel>` component that opens as a right-side drawer from the canvas. Use SSE streaming via `fetchEventSource` or the native `EventSource` API.
- Define a small set of SSE event types: `token`, `tool_call`, `tool_result`, `agent_reasoning`, `error`, `done`. Keep the event schema flat and typed.
- Implement an internal test endpoint that instantiates the agent graph from the current (possibly unsaved) canvas state, avoiding the need to save before testing.
- Display agent reasoning steps in a collapsible section below each response (similar to Flowise's `AgentReasoningCard`).
- Store test conversation history in session storage or a lightweight store, not the database.

## 5. Save/Load

### What It Does
Chatflows are persisted as database records with the full React Flow graph (nodes + edges) serialized as a JSON string. CRUD operations via REST API. Import/export for backup and sharing.

### Key File Paths
| File | Role |
|------|------|
| `packages/server/src/database/entities/ChatFlow.ts` | TypeORM entity: `id`, `name`, `flowData` (JSON text), `type`, `category`, `deployed`, configs |
| `packages/server/src/routes/chatflows/index.ts` | REST routes: GET/POST/PUT/DELETE `/chatflows` |
| `packages/server/src/controllers/chatflows/index.ts` | Request handling, validation |
| `packages/server/src/services/chatflows/index.ts` | Business logic: save, update, delete, file handling |
| `packages/ui/src/api/chatflows.js` | Frontend API client for CRUD calls |
| `packages/ui/src/views/canvas/CanvasHeader.jsx` | Save button, save dialog, export triggers |
| `packages/server/src/routes/export-import/index.ts` | Bulk export/import endpoints |
| `packages/server/src/services/export-import/index.ts` | Export/import logic across entity types |

### Architecture Pattern
- **Single `flowData` column**: The entire graph is stored as a JSON string in one TEXT column. The JSON structure mirrors React Flow's format: `{ nodes: [...], edges: [...] }`. Each node object includes `id`, `data` (with all configured parameter values), `position`, `type`.
- **Metadata columns alongside**: `name`, `type` (CHATFLOW/AGENTFLOW/MULTIAGENT/ASSISTANT), `category`, `deployed`, `isPublic`, `apikeyid`, `chatbotConfig`, `apiConfig`, `analytic`, `speechToText`, `createdDate`, `updatedDate`.
- **Dirty tracking + explicit save**: Canvas tracks an `isDirty` Redux flag. Users must explicitly click Save. The save dialog collects a name, then POSTs/PUTs the full serialized graph.
- **Two-step file save**: If flowData contains base64-encoded file uploads, the first save persists the record, then a second update replaces base64 data with stored file paths.
- **Change detection**: `GET /chatflows/has-changed/{id}/{lastUpdatedDateTime}` lets the UI detect stale data before overwriting.
- **Bulk export/import**: Export serializes multiple entity types (ChatFlow, ChatMessage, Variable, Tool, DocumentStore) into a single JSON payload. Import deserializes and upserts.

### How to Adapt for Archon
- Store the graph as a JSON column (not a TEXT column with manual serialization). Use a typed schema: `{ nodes: AgentNode[], edges: AgentEdge[], viewport: Viewport, metadata: {} }`.
- Add a `version` field to the serialization format for future migration support.
- Implement auto-save with debounce (e.g., 2s after last change) instead of requiring explicit save. Keep a manual save button as well.
- For change detection, use optimistic concurrency: include an `updatedAt` timestamp in update requests and reject if stale.
- Export format should be self-contained: include node type definitions alongside the graph so imported flows can identify missing node types.
- Consider storing flow snapshots/versions for undo history (Flowise doesn't do this).

## Summary: Key Takeaways for Archon

| Flowise Pattern | Archon Adaptation |
|-----------------|-------------------|
| Server-scanned node registry → Redux | Static manifest or API registry → Zustand |
| Category + tag palette with fuzzy search | Category palette with `fuse.js` search |
| React Flow v11 with Redux + Context | React Flow v12 with Zustand + `useReactFlow` |
| Type-driven parameter widgets (if-else chain) | Type-driven widgets via registry map |
| Dual SSE/HTTP prediction endpoints | SSE-first test endpoint with agent graph instantiation |
| JSON-in-TEXT-column persistence | Typed JSON column with version field + auto-save |
| Explicit save with dirty tracking | Auto-save + manual save + optimistic concurrency |
| No undo/versioning | Flow version snapshots for undo |
