# Idun Agent Platform — Architecture Research

> Source: [github.com/Idun-Group/idun-agent-platform](https://github.com/Idun-Group/idun-agent-platform)
> Analyzed: 2025-07-24

## Platform Overview

Idun is a 3-tier agent governance platform:

| Tier | Technology | Purpose |
|------|-----------|---------|
| **Manager** | FastAPI + PostgreSQL | Config CRUD, signed configs, SSO/RBAC, tenancy |
| **Engine** | FastAPI + LangGraph/ADK | Agent runtime with memory, guardrails, tracing, MCP |
| **UI** | Next.js | Admin dashboard for agent governance |

Key repos structure:
- `services/idun_agent_manager/` — Control plane API
- `libs/idun_agent_engine/` — Agent runtime engine
- `libs/idun_agent_schema/` — Shared Pydantic models (schema-first design)
- `services/idun_agent_web/` — Admin UI

## 1. Guardrails Integration

### What It Does

Pre/post-processing safety pipeline that validates every agent input and output against configurable guardrail rules. Supports 14+ guardrail types including PII detection, toxic language, jailbreak detection, bias checking, and custom LLM-based validators.

### Key File Paths

| File | Role |
|------|------|
| `libs/idun_agent_schema/.../engine/guardrails_v2.py` | 14 config schemas using Pydantic discriminated unions |
| `libs/idun_agent_engine/.../guardrails/base.py` | Abstract `BaseGuardrail` with `validate()` interface |
| `libs/idun_agent_engine/.../guardrails/guardrails_hub/guardrails_hub.py` | Hub integration — Strategy pattern with lazy init |
| `libs/idun_agent_engine/.../server/routers/agent.py` | Runtime invocation — calls `_run_guardrails()` pre/post |
| `libs/idun_agent_engine/.../lifespan.py` | Guard parsing & initialization at app startup |
| `services/idun_agent_manager/routers/guardrails.py` | CRUD API for guardrail configs |

### Architecture Pattern

**Strategy + Middleware Pipeline**

```
Request → Input Guardrails (filter position="input") → Agent Execution → Output Guardrails (filter position="output") → Response
           │                                                                │
           └→ Fail? HTTP 429 + reject_message                              └→ Fail? HTTP 429 + reject_message
```

1. **Schema layer**: `GuardrailsV2` defines separate `input: list[GuardrailConfig]` and `output: list[GuardrailConfig]` fields. Each config is a discriminated union type with guard-specific parameters.
2. **Factory**: `get_guard_instance()` maps config IDs → concrete guard classes (BanList, DetectPII, etc.).
3. **Lazy init**: Guards are installed/configured once at startup via `setup_guard()` using the Guardrails AI library.
4. **Runtime**: Each guard stores `self.position` ("input" or "output"). `_run_guardrails()` filters by position, calls `guard.validate(text)`, rejects with HTTP 429 on failure.
5. **Config flow**: Manager stores configs in PostgreSQL → Engine fetches via API → `_parse_guardrails()` hydrates guard instances at startup.

### How to Adapt for Archon

- **Adopt the pipeline pattern**: Wrap agent invocation with pre/post guardrail hooks rather than embedding safety into agent logic. Separation of concerns is key.
- **Use discriminated unions**: Define guardrail configs as a union type so each guard type has its own validated schema. Prevents config errors at parse time.
- **Position tagging**: Tag guards as "input" or "output" at config time, filter at runtime. Simple and effective.
- **Startup init, runtime validate**: Parse and instantiate guards once; call `validate()` per-request. Avoids per-request setup overhead.
- **Reject via HTTP status**: Use HTTP 429 (or 422) for guardrail violations so clients can distinguish safety blocks from errors.

## 2. SSO / RBAC

### What It Does

Currently MVP-level: API key authentication per agent using scrypt-derived keys. Manager endpoints have an optional key check (timing-safe). Full SSO/RBAC via Okta/Auth0 is planned and on the priority roadmap.

### Key File Paths

| File | Role |
|------|------|
| `services/idun_agent_manager/routers/auth.py` | `encrypt_payload()` utility (scrypt-based key derivation) |
| `services/idun_agent_manager/deps.py` | `allow_user()` dependency — `secrets.compare_digest()` check |
| `services/idun_agent_manager/routers/agents.py` | Agent API key generation via `/key` endpoint |
| `.env.example` | Auth config templates (OIDC fields prepared) |
| `ROADMAP.md` | SSO/RBAC listed as 🔥 priority |

### Architecture Pattern

**API Key + Future OIDC Gateway**

Current state:
1. **Agent auth**: Manager generates API key = `scrypt(f"{agent_id}:{agent_name}", salt=AUTH__SECRET_KEY).hex()`. Stored as `agent_hash` in DB.
2. **Engine auth**: Engine sends `Authorization: Bearer {api_key}` to Manager's `/config` endpoint. Direct string comparison validates.
3. **Manager auth**: Optional `KEY` env var checked via `allow_user()` FastAPI dependency. Currently disabled in MVP.
4. **No RBAC**: No roles, permissions, or tenant isolation. Single-tenant design.

Prepared environment variables for future OIDC:
- `AUTH__PROVIDER_TYPE` (okta)
- `AUTH__CLIENT_ID` / `AUTH__CLIENT_SECRET`
- `AUTH__ISSUER` / `AUTH__SCOPES` / `AUTH__REDIRECT_URI`

### How to Adapt for Archon

- **Start with API key auth** for service-to-service communication. Scrypt derivation from deterministic inputs (agent ID + name + secret) is a clean pattern — no need to store raw keys.
- **Use FastAPI dependency injection** for auth: `Depends(allow_user)` on routes. Easy to swap from API key → JWT → OIDC without changing route handlers.
- **Prepare OIDC config schema early**: Even if not implementing SSO now, define the config fields so the migration path is clear.
- **Timing-safe comparison**: Always use `secrets.compare_digest()` for token validation to prevent timing attacks.
- **Plan for multi-tenancy**: Add `tenant_id` to core models from the start, even if you only have one tenant. Adding it later requires migrations and data backfill.

## 3. MCP Tool Control

### What It Does

Manages MCP server connections via a pluggable transport layer (4 transport types) with a centralized registry. Tools from MCP servers are discovered at runtime and bound to LLM agents via LangChain tool adapters.

### Key File Paths

| File | Role |
|------|------|
| `libs/idun_agent_schema/.../engine/mcp_server.py` | `MCPServer` Pydantic model with transport validation |
| `libs/idun_agent_engine/.../mcp/registry.py` | `MCPClientRegistry` wrapping `MultiServerMCPClient` |
| `libs/idun_agent_engine/.../mcp/helpers.py` | Transport utilities and tool loading |
| `services/idun_agent_manager/routers/mcp_servers.py` | CRUD API for MCP server configs |
| `libs/idun_agent_engine/examples/09_langgraph_mcp/` | Integration examples |

### Architecture Pattern

**Two-Tier Registry with Transport Abstraction**

```
Manager (PostgreSQL) → API → Engine Config → MCPClientRegistry → MultiServerMCPClient
                                                                        │
                                                          ┌─────────────┼─────────────┐
                                                        stdio         sse      streamable_http
                                                     (subprocess)  (push SSE)    (HTTP stream)
```

1. **Schema validation**: `MCPServer` model uses `_validate_transport_fields()` post-validator to enforce required fields per transport (e.g., `command` + `args` for stdio, `url` for HTTP types).
2. **Registry**: `MCPClientRegistry` holds list of `MCPServer` configs, creates `MultiServerMCPClient` (from `langchain_mcp_adapters`), exposes `get_tools(server_name)`.
3. **Transport normalization**: `as_connection_dict()` converts transport-specific params into unified dict (e.g., timeouts → `timedelta` for streamable_http, encoding for stdio).
4. **Tool discovery**: `get_langchain_tools()` loads config, initializes registry, returns LangChain-compatible tool objects that the LLM can bind and invoke.
5. **Config priority**: Explicit path → `IDUN_CONFIG_PATH` env var → API fetch from Manager (requires `IDUN_AGENT_API_KEY`).

**Transport types**: stdio (local subprocess), sse (server-sent events), streamable_http (HTTP with streaming), websocket (bidirectional).

### Security Gaps (Instructive)

- No tool allowlisting per agent — all configured tools are available to all agents.
- No parameter validation/filtering at invocation time.
- No rate limiting or audit logging at the tool layer.
- Sandboxing is OS/network-level only (stdio subprocess permissions, HTTP endpoint network isolation).

### How to Adapt for Archon

- **Central MCP registry**: Store approved MCP server configs in a database, not scattered config files. CRUD API enables governance.
- **Transport abstraction**: Use a model like `MCPServer` that normalizes transport differences. `as_connection_dict()` is a clean adapter pattern.
- **Per-agent tool scoping**: Idun's gap — fix this by adding a `tools: list[str]` field to agent configs that allowlists specific tools from specific servers.
- **Schema-first validation**: Pydantic post-validators that enforce transport-specific required fields prevent runtime errors from misconfiguration.
- **Lazy tool loading**: Discover tools at agent invocation time, not at global startup. This allows per-agent tool scoping and dynamic MCP server changes.

## 4. OpenTelemetry / Observability

### What It Does

Pluggable observability system with 5+ provider backends. Supports both LangChain callback-based tracing (Langfuse, LangSmith) and OpenTelemetry SDK-based tracing (GCP Trace, Arize Phoenix). Separate telemetry system for platform usage metrics via PostHog.

### Key File Paths

| File | Role |
|------|------|
| `libs/idun_agent_engine/.../observability/handler_base.py` | Abstract `ObservabilityHandlerBase` |
| `libs/idun_agent_engine/.../observability/langfuse_handler.py` | Langfuse exporter (callback-based) |
| `libs/idun_agent_engine/.../observability/phoenix_handler.py` | Arize Phoenix tracer (OTel-based) |
| `libs/idun_agent_engine/.../observability/gcp_trace_handler.py` | GCP Cloud Trace (OTel SDK) |
| `libs/idun_agent_engine/.../observability/factory.py` | `create_observability_handler()` factory |
| `libs/idun_agent_schema/.../engine/observability_v2.py` | Provider config schemas |
| `libs/idun_agent_engine/.../telemetry/config.py` | PostHog usage telemetry (opt-out) |
| `services/idun_agent_manager/routers/observability.py` | CRUD API for observability configs |

### Architecture Pattern

**Abstract Factory + Provider Strategy**

```
ObservabilityConfig (provider enum + config union)
        │
        ▼
create_observability_handler() ─── Factory
        │
        ├── LangfuseHandler       → LangChain CallbackHandler
        ├── PhoenixHandler         → OpenTelemetry TracerProvider + LangChainInstrumentor
        ├── GCPTraceHandler        → OTel SDK + BatchSpanProcessor + GuardrailsInstrumentor + MCPInstrumentor
        ├── GCPLoggingHandler      → Cloud Logging structured logs
        └── LangSmithHandler       → LangSmith tracing
```

1. **Handler abstraction**: Each handler implements `get_callbacks()` (returns LangChain callbacks or empty list for OTel providers) and `get_run_name()`.
2. **Two instrumentation paradigms**:
   - **Callback-based** (Langfuse, LangSmith): Returns LangChain `CallbackHandler` objects injected into agent invoke/stream calls.
   - **OTel-based** (Phoenix, GCP Trace): Registers global `TracerProvider` + instrumentors. No callbacks needed — tracing is automatic.
3. **Cross-cutting instrumentation**: GCP Trace handler additionally instruments Guardrails, VertexAI, and MCP operations via `GuardrailsInstrumentor`, `VertexAIInstrumentor`, `MCPInstrumentor`.
4. **Config normalization**: Factory handles both V1 (legacy) and V2 schemas uniformly.
5. **Lazy dependencies**: Provider-specific libraries imported only when that provider is selected. Missing deps don't crash the system.

**Telemetry vs Observability distinction**:
- **Telemetry** (`telemetry/`): Platform usage metrics → PostHog. Non-blocking background thread. Opt-out via `IDUN_TELEMETRY_ENABLED=false`. Sanitizes sensitive data. Persistent `distinct_id` in OS cache.
- **Observability** (`observability/`): Agent execution tracing → user-chosen backend. Captures LLM calls, guardrail decisions, tool invocations as spans.

### How to Adapt for Archon

- **Abstract handler pattern**: Define a minimal interface (`get_callbacks()`, `get_run_name()`) that all providers implement. Swap backends via config, not code changes.
- **Support both paradigms**: LangChain callbacks for some providers, OpenTelemetry SDK for others. The factory pattern makes this transparent to the agent runtime.
- **Instrument cross-cutting concerns**: GCP Trace handler instrumenting guardrails and MCP separately is powerful. Add instrumentors for each subsystem rather than one monolithic tracer.
- **Lazy imports**: Only import provider SDKs when needed. This keeps the core lightweight and avoids dependency conflicts.
- **Separate telemetry from observability**: Platform usage metrics (anonymous, aggregated) are different from agent execution traces (detailed, per-request). Keep them in separate systems.
- **Config-driven provider selection**: Store observability config per-agent in the database. Allow different agents to use different tracing backends if needed.

## Cross-Cutting Patterns Worth Adopting

### Schema-First Design
Idun shares Pydantic models (`idun_agent_schema`) across Manager and Engine with zero conversion layers. This eliminates serialization bugs and ensures config validated at parse time.

### Config Flow: Manager → Engine
Centralized config in PostgreSQL, fetched by Engine at startup via authenticated API. Config priority: explicit path → env var → API fetch. Enables governance without coupling services.

### Startup Init, Runtime Execute
Guards, MCP registries, and observability handlers are all initialized once during app `lifespan` startup. Runtime paths only call `validate()`, `get_tools()`, or `get_callbacks()`. Minimizes per-request overhead.

### CRUD-First Governance
Every subsystem (guardrails, MCP servers, observability) has a CRUD API in the Manager. This creates an audit trail and enables UI-driven governance without config file sprawl.

### Graceful Degradation
Missing optional dependencies don't crash the system. Disabled providers return no-op handlers. Guardrails can be disabled globally. Telemetry is opt-out.
